## 1. 프로젝트 개요

### 1.1 배경 및 목적

스마트 창고 운영 환경에서는 로봇 상태, 주문 유입량, 레이아웃 구조, 환경 조건 등 수십 가지 변수가 복합적으로 작용하여 출고 지연을 유발한다. 이러한 지연은 고객 불만, 물류 비용 증가, 운영 효율 저하로 직결된다.

본 프로젝트는 **다음 30분 평균 출고 지연 시간**을 사전 예측함으로써 운영자가 선제적으로 대응할 수 있는 예측 모델을 구축하는 것을 목표로 한다.

### 1.2 문제 정의

| 항목 | 내용 |
|------|------|
| 문제 유형 | 지도학습 / 회귀(Regression) |
| 입력 데이터 | 창고 운영 상태 피처 93개 + 레이아웃 정보 14개 |
| 예측 대상(Target) | `avg_delay_minutes_next_30m` — 다음 30분 평균 출고 지연 시간(분) |
| 평가 지표 | RMSE (Root Mean Squared Error) |

## 2. 데이터 현황

### 2.1 데이터셋 구성

| 파일명 | 행 수 | 컬럼 수 | 비고 |
|--------|-------|---------|------|
| `train.csv` | 250,000 | 94 | 학습 데이터 (타겟 포함) |

### 2.2 데이터 구조

- **시계열 단위**: 15분 간격, 24 타임스텝(6시간) = 1 시나리오
- **시나리오 수**: 약 12,000개 (`scenario_id: SC_00000 ~ SC_05114`)
- **창고 수**: 300개 (`layout_id: WH_001 ~ WH_300`)

### 2.4 레이아웃 유형

| 유형 | 특징 |
|------|------|
| `narrow` | 좁은 통로, 밀집 구조 |
| `grid` | 격자형, 균일 동선 |
| `hybrid` | 혼합형 |
| `hub_spoke` | 중앙 허브 중심 방사형 구조 |

### 2.5 데이터 품질 이슈

- 다수의 결측값(NA) 존재 → 결측 처리 전략 수립 필요
- 피처 값의 스케일이 상이 → 스케일 정규화 고려
- 시계열 구조로 인해 단순 KFold 사용 시 데이터 누수(leakage) 위험 존재

## 3. 모델링 전략

### 3.1 검증 전략

- **Group K-Fold (k=5)** 적용
- `scenario_id` 기준으로 그룹핑하여, **동일 시나리오 내 타임스텝이 train/valid에 동시 포함되는 것을 방지**
- 목적: 데이터 누수 없이 모델의 시나리오 일반화 성능 측정

### 3.2 3단계 접근법 (Phase별 전략)

#### Phase 1 — GBDT Baseline (로컬 환경)

| 항목 | 내용 |
|------|------|
| 환경 | MacBook (Apple Silicon CPU 최적화) |
| 모델 | LightGBM + XGBoost + CatBoost |
| 검증 | Group K-Fold OOF 예측 |
| 목적 | 기준 성능 확보 및 피처 중요도 파악 |

**LightGBM 베이스라인 주요 하이퍼파라미터**:
```
n_estimators: 1000 | learning_rate: 0.05 | max_depth: 7
num_leaves: 63 | subsample: 0.8 | colsample_bytree: 0.8
reg_alpha: 0.1 | reg_lambda: 0.1 | early_stopping: 50
```

#### Phase 2 — 피처 엔지니어링 강화 (로컬 환경)

시계열 특성을 반영한 피처 생성으로 모델 성능 향상:

| 피처 유형 | 설명 |
|---------|------|
| Lag 피처 | 동일 시나리오 내 t-1, t-2, t-3 타임스텝의 타겟값 |
| Rolling 통계량 | 3/5 스텝 이동 평균 및 표준편차 |
| 누적 지표 | 시나리오 내 누적 결함 수, 누적 대기 시간 등 |
| 교차 피처 | `layout_id × time_step` 등 레이아웃-시간 상호작용 |
| 레이아웃 결합 | `layout_info.csv` merge — `layout_type`, `robot_total`, `floor_area_sqm` 등 활용 |

#### Phase 3 — 딥러닝 모델 & 스태킹 앙상블 (클라우드 환경)

| 항목 | 내용 |
|------|------|
| 환경 | Google Colab T4 (GPU) |
| 모델 | TabNet, MLP, LSTM, Transformer |
| 최종 앙상블 | GBDT OOF + DL OOF → 메타 모델 (Linear/Ridge) |

### 3.3 하이퍼파라미터 최적화

- **Optuna** 기반 베이지안 최적화 적용
- 각 모델별 독립적 탐색 후, 앙상블 가중치 최적화

## 4. 성능 향상 로드맵

| 단계 | 작업 | 예상 RMSE 개선 |
|------|------|--------------|
| Step 1 | GBDT Baseline 구축 (LightGBM 단일 모델) | 기준점 |
| Step 2 | layout_info 결합 + 범주형 인코딩 | -3% ~ 5% |
| Step 3 | Group K-Fold로 검증 전략 개선 | 신뢰도 향상 |
| Step 4 | Lag/Rolling 시계열 피처 추가 | -5% ~ 15% |
| Step 5 | XGBoost + CatBoost 앙상블 | -2% ~ 5% |
| Step 6 | Optuna 하이퍼파라미터 튜닝 | -2% ~ 5% |
| Step 7 | 딥러닝 모델 스태킹 (LSTM, TabNet) | -3% ~ 8% |

## 7. 리스크 및 대응 방안

| 리스크 | 대응 방안 |
|--------|---------|
| 데이터 누수 (Leakage) | Group K-Fold 적용, scenario_id 기준 분할 |
| 과적합 | Early Stopping, 정규화(reg_alpha/lambda), Dropout |
| 결측값 처리 오류 | 피처별 결측 패턴 분석 후 전략적 imputation |
| 클라우드 비용 | Phase 1~2를 로컬에서 완료 후 Phase 3 진행 |
| 시계열 구조 무시 | Lag/Rolling 피처 생성으로 시계열 정보 보전 |
